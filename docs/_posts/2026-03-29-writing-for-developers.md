很多技术文章的真正问题，不是内容不对，而是组织方式不够友好。

写给开发者看时，我通常会刻意检查三件事：

## 先给方向，再讲细节

如果一开始就进入实现，很容易让读者不知道自己正在解决什么问题。先交代背景、目标和边界，会让后面的内容更容易被吸收。

## 把抽象概念落到例子里

抽象定义需要存在，但不要停在定义上。一个简短、具体、真实的例子，往往能替代很多解释。

## 代码也需要排版

代码块不仅要对，还要易读。比如这段样式：

```css
.article-body pre {
  overflow-x: auto;
  padding: 1.25rem;
  border-radius: 1rem;
  background: var(--surface-elevated);
}
```

当代码块有舒服的边距、对比和节奏，读者会更愿意继续往下读。阅读体验不是附属项，它本身就是内容质量的一部分。
